自动拆箱和装箱

计算数值时，包装类自动转换成基本数据类型进行计算 —-自动拆箱
当将基本数据类型传给包装类时，基本数据类型自动包装成包装类 —— 自动装箱

原理

自动拆箱和装箱的原理与常量池有关，其实现方式和String的intern()同理

自动拆箱陷阱

自动拆箱和装箱也是以常量池为基础的，但是基本数据类型的常量池是-128~127，在这个范围的基本数据类型的包装类可以自动拆箱，比较时直接进行数值比较。但是如果是不是这个范围内的数值比较（==），则有可能会出现false的情况，因为进行的是引用地址的比较。

String及其包装类

发表于 2019-03-11 | 分类于 java基础

String

String是不可变的

  • String类是final修饰，表明其不可以被继承，即不可能通过子类重写其方法
  • 内部数组也是final修饰，一旦赋值不可更改其引用
  • 内部数组字段value被private修饰，且内部没有public方法修改内部数组，无法通过外部方法修改数组中的值

String不可变性有什么好处

  1. 字符串常量池的需要。字符串常量池的诞生是为了提升效率和减少内存分配。可以说我们编程有百分之八十的时间在处理字符串，而处理的字符串中有很大概率会出现重复的情况。正因为String的不可变性，常量池很容易被管理和优化。

  2. 安全性考虑。正因为使用字符串的场景如此之多，所以设计成不可变可以有效的防止字符串被有意或者无意的篡改。从java源码中String的设计中我们不难发现，该类被final修饰，同时所有的属性都被final修饰，在源码中也未暴露任何成员变量的修改方法。（当然如果我们想，通过反射或者Unsafe直接操作内存的手段也可以实现对所谓不可变String的修改）。

  3. 作为HashMap、HashTable等hash型数据key的必要。因为不可变的设计，jvm底层很容易在缓存String对象的时候缓存其hashcode，这样在执行效率上会大大提升。

字符串常量池

字符串常量池的优点就是避免了字符串重复的创建，节省资源，提升性能和效率

过程

每当创建字符串时，jvm首先检查字符串常量池，如果常量池中存在该字符串，就直接返回该字符串在常量池中的引用。如果不存在，就会实例化该字符串并将其放置到常量池中

String的连接

‘+’的本质是编译的时候将‘+’替换成StringBuilder.append()

  • 使用只包含常量的字符串连接符创建的也是常量，编译期就能确定，一经确定就存储到常量池中。而且常量间的相加因为编译器会优化的原因，不会引起效率问题
  • 使用包含变量的字符串连接符创建的对象只有到运行期才能够确定并创建的，存储在堆中。

intern()

不同版本的intern实现方式不同:

  • jdk6中的intern()方法会把首次出现的字符串实例复制到常量池中，返回的也是该实例在常量池中的地址。
  • jdk7中intern()方法不会复制实例，只是在常量池中记录首次出现的实例引用，返回的是堆中的对象引用

equals()

作为重写基类Object的方法，String.equals()方法能够判断其内容是否相同

总结： equals方法的实现原理:显示判断两者的地址是否相等;判断是否是String类型;判断数组的长度是否相等;遍历，挨个判断数组中的元素是否相等。

equals和==的区别

对于==，如果作用于基本数据类型的变量，则直接比较其存储的值是否相等；如果作用的是引用类型的变量，则是比较它们所指向的地址。
equals是基类中的方法，因此所有引用对象都有这个方法，在Object类中是用来比较两个对象的引用是否相等，即是否指向的是同一个对象。
由于String重写了基类的equals方法，所以比较的是两个字符串对象的内容是否相等。
对于equals方法，equals方法不能作用于基本数据类型的变量，因为上述原因，比较的是变量地址而已。

StringBuilder和StringBuilder

它们的底层同样是继承了父类AbstractStringBuilder的可变字符数组.

扩容

扩容策略：当新字符串的长度大于value数组的长度则进行扩容，扩容后的长度是原来的两倍+2；一般来说扩容的上限是jvm支持的最大数组长度MAX_ARRAY_SIZE。如果这个最大数组长度大于int的最大值，会抛出异常（当新的字符串超过int最大值时）；否则以数组最大长度作为新数组的长度。

delete 删除

这两个类的删除操作都是调用父类的delete方法进行删除。事实上是将剩余的字符重新拷贝到字符数组value。

String、StringBuilder、StringBuffer的区别

  • 可变与不可变：String是不可变字符串对象，StringBuilder和StringBuffer是可变字符串对象
  • 是否线程安全：String和StringBuffer线程安全，StringBuilder线程不安全。String中的对象不可变性，可以理解为常量。StringBuffer虽然实现方式和StringBuilder相同且继承于同一个父类AbstractStringBuilder，但是StringBuffer的方法大多使用了Synchronized关键字进行修饰，因此线程安全。
  • 执行效率：StringBuilder>StringBuffer>String 但是这个执行效率是相对的，不一定在所有的情况下都这样。
  • 适合的场景：String适合于少量的字符串操作；StringBuilder适合于单线程下在字符缓冲区进行大量操作；StringBuilder适用于多线程下字符缓冲区进行大量操作的情况。

java oop基础

发表于 2019-03-11 | 分类于 java基础

oop的特点

oop（面向对象编程）引入了封装、继承和多态等概念来建立一种对象层次结构，用以模拟公共行为的一个集合。

封装

隐藏了类的内部实现机制，可以在不影响使用的情况下改变类的内部结构，同时也保护了数据。对外界而言它的内部细节是隐藏的，暴露给外界的只是它的访问方法。

java当中的封装主要体现在Java的访问权限控制：public >protect>default=package>private。封装可以保护类中的信息，只提供外界它的访问方法。

访问关键字的范围

继承

继承使得子类具有父类的属性和方法或者方法的重新定义和扩展

java只支持单继承，但是可以通过内部类继承其他类来实现多继承。

public class Son extend Father{
    public void go(){
        System.out.println("son go");
    }

    public void eat(){
        System.out.println("son eat");
    }

    public void cook(){
        //匿名内部类实现的多继承
        new Mother().cook();
        //内部类继承第二个父类来实现多继承
        Mom mom=new Mom();
        mom.cook();
    }

    private class Mom extends Mother{
        @Override
        public void cook(){
            System.out.println("mom cook");
        }
    }
}

public class Mother{
    public void cook(){}
}

public class Father{
    public void go(){}
    public void eat(){}
}

多态

对于面向对象而言，多态分为编译时多态和运行时多态。

编译时多态是静态的，主要指的是重载。重载是同一个类中同名方法，可以传入多个参数组合。重载的判定有参数的个数、参数的类型、参数的顺序。参数的返回值类型不能作为重载的依据，因为编译器并不能识别它们，导致编译出错。重载在编译的时候就能够根据传入的参数组合，决定调用的方法了。
运行时多态是重写，重写是继承关系的子类有和父类同名同参数的方法，覆盖父类的实现。因此重写方法在编译时并不能直到子类调用的时哪个方法，但是运行时操作数栈会根据子类的引用去子类的类信息中查找方法，找不到时再去父类的类信息中查找方法。
编译器的静态分派和运行期的动态分派对应的场景就是它们（重载和重写）

向上转型和向下转型

public static void main(String[] args) {
    Son son = new Son();
//首先先明确一点，转型指的是左侧引用的改变。
//father引用类型是Father，指向Son实例，就是向上转型，既可以使用子类的方法，也可以使用父类的方法。
//向上转型,此时运行father的方法
    Father father = son;
    father.smoke();
//不能使用子类独有的方法。
// father.play();编译会报错
    father.drive();
//Son类型的引用指向Father的实例，所以是向下转型，不能使用子类非重写的方法，可以使用父类的方法。

//向下转型，此时运行了son的方法
    Son son1 = (Son) father;
//转型后就是一个正常的Son实例
    son1.play();
    son1.drive();
    son1.smoke();

//因为向下转型之前必须先经历向上转型。
//在向下转型过程中，分为两种情况：

//情况一：如果父类引用的对象如果引用的是指向的子类对象，
//那么在向下转型的过程中是安全的。也就是编译是不会出错误的。
//因为运行期Son实例确实有这些方法
    Father f1 = new Son();
    Son s1 = (Son) f1;
    s1.smoke();
    s1.drive();
    s1.play();

//情况二：如果父类引用的对象是父类本身，那么在向下转型的过程中是不安全的，编译不会出错，
//但是运行时会出现java.lang.ClassCastException错误。它可以使用instanceof来避免出错此类错误。
//因为运行期Father实例并没有这些方法。
    Father f2 = new Father();
    Son s2 = (Son) f2;
    s2.drive();
    s2.smoke();
    s2.play();

//向下转型和向上转型的应用，有些人觉得这个操作没意义，何必先向上转型再向下转型呢，不是多此一举么。其实可以用于方法参数中的类型聚合，然后具体操作再进行分解。
//比如add方法用List引用类型作为参数传入，传入具体类时经历了向下转型
    add(new LinkedList());
    add(new ArrayList());

}

public static void add(List list) {
    System.out.println(list);
    //在操作具体集合时又经历了向上转型
    ArrayList arr = (ArrayList) list;
    LinkedList link = (LinkedList) list;
}

总结：
向上转型和向下转型都是针对引用的转型，是编译期进行的转型，根据引用类型来判断使用哪个方法。并且在传入方法时会自动进行转型（有需要的话）。运行期将引用指向实例，如果是不安全的转型则会报错，若安全则继续执行方法，即编译时不报错，运行时报错。

动态绑定和静态绑定

把一个方法与其所在的类/对象关联起来叫做方法的绑定。绑定分为静态绑定（前期绑定）和动态绑定（后期绑定）。
静态绑定（前期绑定）是指：在程序运行前就已经知道方法是属于那个类的，在编译的时候就可以连接到类中，定位到这个方法。
在Java中，final、private、static修饰的方法以及构造函数都是静态绑定的，不需程序运行，不需具体的实例对象就可以知道这个方法的具体内容。
动态绑定（后期绑定）是指：在程序运行过程中，根据具体的实例对象才能具体确定是哪个方法。
动态绑定是多态性得以实现的重要因素，它通过方法表来实现：每个类被加载到虚拟机时，在方法区保存元数据，其中，包括一个叫做 方法表（method table）的东西，表中记录了这个类定义的方法的指针，每个表项指向一个具体的方法代码。如果这个类重写了父类中的某个方法，则对应表项指向新的代码实现处。从父类继承来的方法位于子类定义的方法的前面。

动态绑定的编译、运行原理：

编译
 在代码的编译阶段，编译器通过引用类型 在方法区中该类型的方法表中查找匹配的方法（最佳匹配法：参数类型最接近的被调用），如果有则编译通过。
编译阶段是确保方法的存在性，保证程序能顺利、安全运行。
运行
编译阶段在声明对象类型的方法表中查找方法，只是为了安全地通过编译（也为了检验方法是否是存在的）。
而在实际运行这条语句时，在执行 Father ft=new Son(); 这一句时创建了一个Son实例对象，然后在 ft.say() 调用方法时，JVM会把刚才的son对象压入操作数栈，用它来进行调用。
而用实例对象进行方法调用的过程就是动态绑定：根据实例对象所属的类型去查找它的方法表，找到匹配的方法进行调用。若无重写，则按照父类中的方法表顺序保存在子类方法表中。故此：动态绑定根据对象的类型的方法表查找方法是一定会匹配
# 静态分派
在编译期间，可以根据变量的静态类型确定方法执行版本的分派动作称作静态分派。典型的是方法的重载。
# 动态分派
在编译期间确定不下来，只有在方法调用时，根据对象的实际类型确定方法执行版本的分派动作称作动态分派。 重写。
动态分派的实现：
类在方法区中有一个方法表，方法表中存放着各个方法的实际入口地址，如果某个方法在子类没有被重写，那么子类的虚方法表里面的地址入口和父类相同方法的地址入口是一致的，都指向父类的实现入口。如果子类中重写了这个方法，子类方法表中的地址将会替换为指向子类的实现入口。
